# Core 架构

这份文档只讲 Core 最重要的运行方式。

## 一句话

> Core 负责统一装配，插件负责具体平台能力。

## 先记住 4 个词

| 词 | 含义 |
| --- | --- |
| `PluginInfo.ID` | 插件运行时唯一标识 |
| `platform` | 业务平台键 |
| `display_name` | UI 展示名 |
| `plugin.yaml` | 分发产物，不是运行时真相 |

## Core 和插件怎么分工

Core 负责：

- 用户、账号、分组、API Key
- 调度、限流、并发、计费
- 插件进程管理
- 插件路由注册
- 插件前端资源挂载

插件负责：

- 和上游平台通信
- 声明自己的路由、模型、账号类型
- 提供自己的前端页面或账号表单

## 一个请求怎么走

1. 客户端请求 Core 暴露的接口
2. Core 根据 `platform` 选择合适账号
3. Core 做限流、并发控制、计费准备
4. Core 调用插件转发请求
5. 插件和上游平台通信
6. Core 记录 usage 并返回结果

## 一个插件怎么接进来

1. Core 启动插件进程
2. 通过 gRPC 读取插件元信息
3. Core 用 `PluginInfo.ID` 注册插件实例、路由和资源路径
4. Core 用 `platform` 把插件接入账号、调度和计费链路
5. 管理后台显示插件信息和账号表单

## 前端插件怎么接进来

1. 插件提供前端静态资源
2. Core 把资源挂到 `/plugins/{name}/assets/*`
3. 管理后台动态加载插件模块
4. 插件渲染自己的页面或账号表单

## 为什么要区分 `PluginInfo.ID` 和 `platform`

- `PluginInfo.ID`
  - 是运行时键
  - 用在 API、资源路径、动态加载、内部缓存
- `platform`
  - 是业务键
  - 用在账号、调度、计费、使用记录

这两个值可以相同，也可以不同，但语义不能混用。

## 为什么 `plugin.yaml` 不是运行时真相

因为运行时真正被 Core 消费的是插件进程通过 gRPC 返回的元信息。

`plugin.yaml` 只用于：

- 安装
- 发布
- 市场展示
- 版本兼容检查
